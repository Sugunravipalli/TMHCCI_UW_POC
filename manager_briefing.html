1Executive Summary

We will build a medallion architecture on Snowflake where raw underwriting artifacts flow from the stage, get parsed by Cortex AI into a curated Silver layer, then fork into TWO parallel and independent schemas: a Graph schema that hosts the ontology as the prime decision engine for rules, constraints, and reasoning — and an Analytics schema that hosts semantic views for Cortex Analyst-powered metrics and dashboards. A Cortex Agent orchestrates both, routing questions by intent.

Key Architectural Principle

The Graph and Analytics schemas are completely independent at query time. Neither reads from the other. They are unified only by sharing canonical IDs from the Curated layer — this is the single integration contract. Each schema can evolve on its own schedule, be owned by a different team, and be optimized for its own class of questions. The Cortex Agent sees them as two separate tools and routes between them by intent.

Risk Network Example — "Northeast Flood Exposure"

The kind of connection discovery Neo4j makes trivial. Ask: "Show me every policy whose location sits in a flood-risk zone." In Snowflake this is a multi-join query. In Neo4j it's a single pattern match.

// Find every customer with a policy exposed to flood risk MATCH (c:Customer)-[:SUBMITS]->(:Submission) -[:BOUND_AS]->(p:Policy) -[:AT_LOCATION]->(loc:Location) -[:HAS_RISK]->(r:RiskZone {type: 'FLOOD'}) WHERE loc.region = 'Northeast' RETURN c.name, p.policy_number, loc.city, r.severity, r.fema_zone ORDER BY r.severity DESC;

Neo4j Bloom then renders the matching pattern as a live, clickable graph — the underwriter can expand any customer to see their full history, brokers, prior claims, adjacent risks.

Implementation — Four Concrete Steps

1 Deploy Neo4j (SPCS or Aura)
For SPCS: Register a compute pool, upload Neo4j image, create service using this spec:
CREATE SERVICE NEO4J_VIZ IN COMPUTE POOL POOL_NEO4J FROM SPECIFICATION $$ spec: containers: - name: neo4j image: /db_uw/images/neo4j:5-community env: NEO4J_AUTH: neo4j/$SECRET_PWD volumeMounts: - name: data mountPath: /data endpoints: - name: bolt port: 7687 public: true - name: http port: 7474 public: true $$;
For Aura: One-click provision on console.neo4j.io, copy connection string.
2 Sync from Snowflake GRAPH → Neo4j
Use a Snowflake Task running every N minutes that reads new rows from KG_NODE and KG_EDGE and pushes them to Neo4j via Bolt. Or use the official Neo4j Connector for Snowflake Native App which handles it declaratively.
-- Snowflake Task example CREATE TASK SYNC_TO_NEO4J WAREHOUSE = WH_UW SCHEDULE = '15 MINUTES' AS CALL DB_UW.GRAPH.EXPORT_TO_NEO4J();
3 Materialize Nodes & Relationships in Neo4j
The sync procedure translates each KG_NODE row into a Cypher MERGE statement, and each KG_EDGE row into a relationship MERGE.
// For each KG_NODE row: MERGE (n:Customer {node_id: 'CUST_0001'}) SET n.name = 'Acme Corporation', n.naics = '332710', n.state = 'IL'; // For each KG_EDGE row: MATCH (a {node_id: 'CUST_0001'}) MATCH (b {node_id: 'SUB_2026_0789'}) MERGE (a)-[r:SUBMITS]->(b) SET r.effective_start = '2026-03-15';
4 Expose to Underwriters — Bloom + Streamlit
Neo4j Bloom: Business users get a natural-language search bar — type "flood risk Northeast" and see the whole network.
Streamlit embed: In our Cortex Agent UI, when the answer involves relationships, we render a graph snippet via the pyvis or streamlit-agraph library fed by a Cypher query.
Cortex Agent handoff: The Graph tool description tells the LLM "when the user asks to 'see' or 'show' connections, return a graph URL alongside the text answer."

6⭐ Best Snowflake Feature for Stage → RAW Parsing

🏆 The Winner: SNOWFLAKE.CORTEX.PARSE_DOCUMENT

For unstructured files (emails, PDFs, scanned docs): Cortex PARSE_DOCUMENT is the single best Snowflake-native feature. It reads files directly from the stage, extracts full text + layout + tables + metadata, and returns structured JSON — all without leaving Snowflake. Example: Ingestion Pattern for Emails

-- 1. Discover new email files in stage
SELECT RELATIVE_PATH, SIZE, LAST_MODIFIED
FROM DIRECTORY(@UW_STAGE)
WHERE RELATIVE_PATH LIKE 'emails/%';

-- 2. Parse all of them in one SQL statement using Cortex
CREATE OR REPLACE TABLE DB_UW.RAW.RAW_EMAIL_DOCS AS
SELECT
  RELATIVE_PATH                           AS FILE_NAME,
  SIZE                                    AS FILE_SIZE,
  LAST_MODIFIED                           AS STAGE_MODIFIED_TS,
  SNOWFLAKE.CORTEX.PARSE_DOCUMENT(
    @UW_STAGE,
    RELATIVE_PATH,
    {'mode': 'LAYOUT'}
  )                                       AS PARSED_CONTENT,
  PARSED_CONTENT:content::STRING          AS FULL_TEXT,
  PARSED_CONTENT:metadata                 AS METADATA,
  CURRENT_TIMESTAMP()                     AS _INGEST_TS
FROM DIRECTORY(@UW_STAGE)
WHERE RELATIVE_PATH LIKE 'emails/%';

-- 3. For structured rating sheets — use COPY INTO
COPY INTO DB_UW.RAW.RAW_RATING_ROWS (FILE_NAME, RAW_ROW, _SOURCE_YEAR)
FROM (
  SELECT METADATA$FILENAME, OBJECT_CONSTRUCT(*),
         TO_NUMBER(REGEXP_SUBSTR(METADATA$FILENAME,'policy_(\\d{4})',1,1,'e',1))
  FROM @UW_STAGE/rating/
)
FILE_FORMAT = FF_XLSX
ON_ERROR = 'CONTINUE';

7Ontology as the Prime Engine — Rules & Decisions

Ontology in our architecture is not passive metadata. It is the decision engine. It validates data, encodes business rules, infers new knowledge, and drives what the Cortex Agent does first.

🧠
4 Ways Ontology Becomes Active
From metadata → to intelligence
1. Constraint validation (ONT_CONSTRAINT)
2. Business rule engine (ONT_BUSINESS_RULE)
3. Inference engine (derived edges)
4. Agent tool priority (route ontology first)
Example: "Should we bind Acme Corp's submission?"

Step 1: Graph loads Acme Corp node, traverses prior policies → finds 1 open claim > $100K.

Step 2: Ontology applies rule R001 (prior loss ratio > 0.70 → REFER_TO_UW).

Step 3: Agent queries Analytics for supporting metrics (typical NAICS premium, broker conversion).

Step 4: Response: "REFER_TO_UW — prior loss ratio 0.82 exceeds threshold. Context: broker conversion 62%."

1. Ontology as Constraint Validator

Every class has structural rules the data must obey:

CREATE TABLE DB_UW.GRAPH.ONT_CONSTRAINT (
  CONSTRAINT_ID     STRING,
  SUBJECT_CLASS     STRING,     -- 'Policy'
  RULE_TYPE         STRING,     -- REQUIRED | FORBIDDEN | CARDINALITY
  PREDICATE         STRING,     -- 'RATED_BY'
  OBJECT_CLASS      STRING,     -- 'RatingWorksheet'
  CARDINALITY       STRING,     -- '1..1', '1..N'
  VIOLATION_ACTION  STRING      -- FLAG | BLOCK | WARN
);

INSERT INTO ONT_CONSTRAINT VALUES
  ('C001','Policy','REQUIRED','RATED_BY','RatingWorksheet','1..1','FLAG'),
  ('C002','Policy','REQUIRED','BOUND_FROM','Quote','1..1','BLOCK'),
  ('C003','Claim','REQUIRED','BELONGS_TO','Policy','1..1','BLOCK'),
  ('C004','Submission','REQUIRED','SUBMITTED_BY','Customer','1..1','BLOCK');

2. Ontology as Business Rule Engine

Underwriting decisions encoded alongside the classes they govern:

R001High Loss Ratio Referral
IF Submission.customer_prior_loss_ratio > 0.70 → REFER_TO_UW
R002Catastrophe Zone Auto-Decline
IF Submission.risk_location IN CAT_ZONES → AUTO_DECLINE
R003Preferred Industry Fast Track
IF Submission.naics IN PREFERRED AND premium < 50K → FAST_TRACK
R004Repeat Claimant Review
IF Customer has >= 3 Claims in past 24 months → SENIOR_UW_REVIEW

3. Ontology as Inference Engine

Stored procedures derive new edges from existing ones — knowledge the raw data never stated explicitly:

-- Rule: Customer → SUBMITS → Submission → QUOTED_AS → Quote → BOUND_AS → Policy
--   implies   Customer → INSURED_BY → Policy
CALL INFER_CUSTOMER_POLICY_EDGES();

-- Rule: Policy → HAS_CLAIM → Claim with loss_date within policy term
--   implies   Claim → COVERED_BY → Policy (validated)
CALL VALIDATE_CLAIM_COVERAGE();

4. Ontology as the Agent's First Tool

Cortex Agent is configured with ontology reasoning at priority 1. The agent tries to understand questions semantically first, then falls through to analytics for numbers:

agent: UW_ASSISTANT
tools:
  - name: graph_reasoning
    priority: 1
    description: Use FIRST for questions about entities, relationships,
                 rules, constraints, violations, or "why" — connections
                 across parties, transactions, events.

  - name: analytics_semantic_view
    priority: 2
    description: Use for numeric aggregations — loss ratios, premiums,
                 conversion rates, counts by dimension, time trends.

  - name: metadata_introspection
    priority: 3

9Identity Resolution — The Silver Integration Contract

The Curated layer's SLV_IDENTITY_RESOLUTION table is the single point where Graph and Analytics schemas share identity. Without it, we can't cross between them.

⚡ Example

Email: "Acme Corporation" · Rating: "ACME CORP" · Claims: "Acme Corp." w/ POL-2024-001 · Promised Lines: "ACME CORPORATION" w/ POL-2024-001

→ Canonical: NODE_ID = CUST_0001 (used in both Graph and Analytics)

Resolution Strategy

  1. Exact match on EIN / Tax ID (highest confidence)
  2. Exact match on Policy Number (links rating → promised lines → claims)
  3. Semantic name matching via Cortex EMBED_TEXT_768 + VECTOR_COSINE_SIMILARITY
  4. Address corroboration (fuzzy match on normalized address)
  5. Assign canonical NODE_ID — written to both GRAPH.KG_NODE and ANALYTICS.DIM_CUSTOMER

13Why This Architecture Wins

✅ Data Sovereignty

Everything stays inside Snowflake. PII, PHI, and financials never leave the governance boundary.

✅ No External AI Stack

PARSE_DOCUMENT, AI_EXTRACT, Cortex Agents replace OpenAI + LangChain + vector DB.

✅ Graph & Analytics Independent

Teams can own them separately. Changes don't cascade. Each optimized for its query class.

✅ Ontology Drives Decisions

Not just metadata — the prime engine for rules, constraints, inference, and "why" questions.

✅ Auditable & Extensible

Medallion layers make every decision traceable. New LOBs = metadata insert, not schema change.

✅ Self-Service for Underwriters

Cortex Agent means no more BI tickets. Answers in seconds, not days.
